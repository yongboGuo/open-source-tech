## 2. 后端总体架构（不是 Agent Runtime，而是 Notes Backend）

MemOS（memos）本质是一个自托管笔记后端，不是 OpenClaw/Hermes 那类“Agent Orchestrator”。

核心拓扑：

1. 启动层（CLI + 配置）
- `cmd/memos/main.go`
- Cobra + Viper 读取参数与环境变量，创建 DB driver，执行 migration，拉起 server。

2. 服务层（单体 Go Server）
- `server/server.go`
- Echo v5 承载 HTTP；注册 frontend/fileserver/RSS/API v1/MCP；启动后台 runner。

3. API 层（双协议）
- `server/router/api/v1/v1.go`
- 同时暴露：
  - gRPC-Gateway 风格 HTTP API（`/api/v1/*`）
  - Connect RPC（`/memos.api.v1.*`）
- 业务逻辑集中在 `APIV1Service`，协议层做适配。

4. 数据访问层（Store + Driver）
- `store/store.go` + `store/driver.go`
- 三套驱动：`store/db/sqlite`、`store/db/mysql`、`store/db/postgres`
- 同一 `FindMemo/UpdateMemo` 等领域模型映射到多数据库 SQL。

5. 扩展接口层（MCP）
- `server/router/mcp/*`
- 内嵌 MCP server，工具调用最终落到现有 APIV1Service / Store。

## 3. 关键执行链路

### 3.1 服务启动链路

`main.go` -> `db.NewDBDriver` -> `store.New` -> `store.Migrate` -> `server.NewServer` -> `server.Start`

关键点：
- migration 在服务启动前执行；
- `server.NewServer` 里统一注册所有路由；
- `server.Start` 后启动 background runners（当前主要是 S3 presign 周期任务）。

### 3.2 API 请求链路（以 ListMemos 为例）

`HTTP/Connect` -> `APIV1Service.ListMemos` -> 组装 `store.FindMemo` -> `Store.ListMemos` -> driver SQL

关键点：
- 过滤语言统一走 CEL 子集（`filter` 字段）；
- 可见性控制在 service 层先做（匿名仅 PUBLIC，登录用户含本人 + PUBLIC/PROTECTED）；
- Store 层只做数据访问，不做业务鉴权。

### 3.3 MCP 工具调用链路（以 create_memo/search_memos 为例）

`/mcp` -> `MCPService` -> tool handler (`tools_memo.go`) -> APIV1Service/Store -> SSE/Webhook side effects

关键点：
- MCP 不是独立后端；只是给 AI 客户端的“北向工具接口”；
- MCP 工具大多复用现有 API 逻辑，避免重复实现。

## 4. 后端模块拆分质量

优点：
- 协议适配与业务逻辑分离（Connect/Gateway 只是壳，APIV1Service 是核心）。
- 多数据库实现通过统一 Store 接口抽象。
- MCP 复用业务层，扩展成本低。
- 迁移体系完整（embed migration + 版本管理 + 事务应用）。

限制：
- 当前仍是单体服务；没有独立任务编排中心。
- AI 能力面较窄（主要是转写）；并非“多 skill runtime”。

## 5. 一句话结论

MemOS 后端是成熟的“内容系统后端 + 多协议 API + MCP 接口”架构；非常适合做“记忆数据底座”，但并不自带复杂 agent 编排引擎。
