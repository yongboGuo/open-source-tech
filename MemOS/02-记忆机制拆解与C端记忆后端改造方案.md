## 1. 先下结论

MemOS 当前“记忆机制”是：

- 以 `memo` 为核心的持久化记忆（SQLite/MySQL/Postgres）
- 以 Markdown 解析产出的 payload（tags/property/mentions）作为结构化记忆索引
- 以关系表（memo_relation）形成记忆关联图
- 以 CEL 过滤 + SQL LIKE/ILIKE 做检索

它可以成为 C 端产品的“长期记忆后端基础层”，但目前还不是 LLM 语义记忆系统（缺 embedding/vector/rerank/回忆策略）。

## 2. 现有记忆机制拆解

### 2.1 长期存储：memo 是主记忆单元

- 主体结构在 `store/memo.go`：`Memo{UID, CreatorID, Content, Visibility, Payload...}`。
- 多数据库持久化在：
  - `store/db/sqlite/memo.go`
  - `store/db/mysql/memo.go`
  - `store/db/postgres/memo.go`

含义：
- 记忆天然持久化；
- 多租户隔离主要依赖 CreatorID + Visibility。

### 2.2 记忆增量更新：写入时实时重建 payload

在 `CreateMemo/UpdateMemo` 中，内容变更会触发：
- `memopayload.RebuildMemoPayload(...)`
- 该函数通过 `markdown.ExtractAll(...)` 一次遍历提取：`tags / mentions / has_link / has_code / has_task_list / has_incomplete_tasks / title`

对应代码：
- `server/router/api/v1/memo_service.go`
- `server/runner/memopayload/runner.go`
- `internal/markdown/markdown.go`

这就是“持续更新记忆索引”的雏形。

### 2.3 记忆关联：relation + comment

- `memo_relation` 用于 REFERENCE/COMMENT 关系。
- 评论也是 memo，本质是父子记忆关系。

价值：
- 具备“记忆图谱”的基础结构，可扩展成事实链路/上下文链路。

### 2.4 检索机制：CEL -> IR -> SQL

`internal/filter` 做了成熟的查询编译管线：
- 解析：CEL 子集
- 归一化：IR
- 渲染：按 sqlite/mysql/postgres 方言输出 SQL

`content.contains("...")` 最终落为 `LIKE/ILIKE`（非向量语义检索）。

## 3. 长期记忆能力评估（按 LLM 记忆标准）

已具备：
- 持久化存储
- 增量更新
- 关系链路
- 权限隔离
- 多数据库支持

缺失：
- embedding 向量索引
- 语义召回与重排
- 记忆分层（短期/长期/用户画像/偏好）
- 记忆置信度、遗忘策略、冲突合并

## 4. 能否做成 C 端记忆后端？

可以，而且很合适做第 1 层底座。

建议采用“保留 memo 主表 + 新增 memory 子系统”的改造方式，而不是推翻重做。

### 4.1 推荐数据模型扩展

新增表（示意）：

1. `memory_chunk`
- `id`, `memo_id`, `user_id`, `chunk_text`, `chunk_type`, `source`, `created_at`, `updated_at`

2. `memory_embedding`
- `chunk_id`, `provider`, `model`, `vector`, `dim`, `version`

3. `memory_fact`
- `id`, `user_id`, `fact_key`, `fact_value`, `confidence`, `source_chunk_id`, `last_verified_at`

4. `memory_profile`
- `user_id`, `preferences_json`, `business_context_json`, `risk_flags_json`

### 4.2 推荐更新流程（持续更新）

1. 写入 memo（已有）
2. 触发事件（Create/Update/MCP mutation）
3. 异步 worker 切分 chunk + embedding
4. 入向量索引（pgvector/外部向量库）
5. 抽取结构化事实并回写 `memory_fact`
6. 下游 agent 检索时做 hybrid recall（keyword + vector + relation）

### 4.3 数据库存储建议

- 如果你偏向简化部署：Postgres + `pgvector`
- 如果要多云和超大规模：向量库外置（Milvus/Weaviate/Qdrant）
- MemOS 原生支持 Postgres，所以迁移到 “Postgres 作为记忆主库” 成本最低。

## 5. 需要额外补齐的能力

- 记忆审计与删除（用户可见、可纠错、可遗忘）
- PII 检测和脱敏
- 记忆版本化（支持回滚）
- 检索质量评估（Recall@K、事实命中率）

## 6. 一句话结论

MemOS 现在是“结构化内容长期存储系统”，离“LLM 长期记忆系统”只差语义索引、召回策略和治理层；改造空间非常现实。
