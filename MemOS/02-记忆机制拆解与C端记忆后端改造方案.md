## TL;DR

- 能做 C 端长期记忆后端：可以。
- 现状完成度：约 70/100（存储与结构化索引很好，语义记忆不足）。
- 最小改造重点：`chunk + embedding + hybrid recall + 记忆治理`。

## 1. 现有“记忆”能力盘点

| 维度 | 现状 | 代码锚点 | 评价 |
|---|---|---|---|
| 长期存储 | `memo` 持久化，三库支持 | `store/memo.go`, `store/db/*/memo.go` | 强 |
| 结构化索引 | `payload` 提取 tags/property/mentions | `memopayload/runner.go`, `internal/markdown/markdown.go` | 强 |
| 关系网络 | `memo_relation` + comment 关系 | `memo_service.go` | 中强 |
| 检索 | CEL -> SQL, `contains` -> LIKE/ILIKE | `internal/filter/*` | 中 |
| 语义召回 | 无 embedding/vector | 全局扫描无向量实现 | 弱 |

## 2. 为什么它适合做 C 端记忆底座

1. 数据模型天然是“用户内容历史”。
2. 读写链路稳定、权限隔离清晰（creator + visibility）。
3. MCP 已提供对 AI 客户端的北向能力入口。
4. 多数据库支持让你迁移成本可控。

## 3. 缺口在哪里（必须补）

1. 向量化索引（embedding + ANN 检索）。
2. 记忆分层（事实、偏好、行为、任务上下文）。
3. 冲突处理与遗忘机制（不是永远累积）。
4. 记忆可审计（可解释、可删除、可更正）。

## 4. 最小可用改造（MVP）

### 4.1 数据模型（建议）

```sql
-- 记忆切片
CREATE TABLE memory_chunk (
  id BIGSERIAL PRIMARY KEY,
  memo_id BIGINT NOT NULL,
  user_id BIGINT NOT NULL,
  chunk_text TEXT NOT NULL,
  chunk_type VARCHAR(32) NOT NULL,
  created_at TIMESTAMP NOT NULL,
  updated_at TIMESTAMP NOT NULL
);

-- 向量索引（以 pgvector 为例）
CREATE TABLE memory_embedding (
  chunk_id BIGINT PRIMARY KEY,
  provider VARCHAR(32) NOT NULL,
  model VARCHAR(64) NOT NULL,
  embedding VECTOR(1536) NOT NULL,
  version INT NOT NULL
);

-- 结构化事实
CREATE TABLE memory_fact (
  id BIGSERIAL PRIMARY KEY,
  user_id BIGINT NOT NULL,
  fact_key VARCHAR(128) NOT NULL,
  fact_value TEXT NOT NULL,
  confidence NUMERIC(4,3) NOT NULL,
  source_chunk_id BIGINT,
  updated_at TIMESTAMP NOT NULL
);
```

### 4.2 写入流水线

1. 用户写入/更新 memo（现有）。
2. 事件落队列（新增）。
3. worker 执行：切 chunk -> embedding -> fact 抽取。
4. 更新索引和事实表。
5. 写审计日志（谁、何时、改了什么）。

### 4.3 检索流水线

1. query 改写（关键词 + 语义查询）。
2. keyword 召回（现有 CEL/SQL）。
3. vector 召回（新增）。
4. relation 扩展（基于 memo_relation）。
5. rerank 与去重。

## 5. 两周落地计划（可执行）

### Week 1

1. 上 Postgres + pgvector。
2. 建 `memory_chunk`、`memory_embedding`。
3. 接入异步任务（先单 worker 即可）。
4. `CreateMemo/UpdateMemo` 写后发任务。

### Week 2

1. 提供 `POST /memory/search`（hybrid）。
2. 为返回结果增加来源解释（memo id、chunk id、score）。
3. 增加删除链路（memo 删除时同步删 chunk/embedding）。
4. 接监控：召回耗时、成功率、队列积压。

## 6. 验收指标

1. Recall@10 >= 基线 +20%。
2. P95 检索延迟 <= 300ms（不含大模型生成）。
3. 写后索引可见延迟 <= 5s。
4. 记忆删除一致性成功率 >= 99.9%。

一句话：MemOS 现在是优秀“记忆存储层”，补上语义索引与治理后即可成为 C 端长期记忆后端。
