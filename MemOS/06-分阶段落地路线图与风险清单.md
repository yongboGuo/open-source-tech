## 1. 目标定义

目标不是“把 MemOS 变成另一个 Agent 平台”，而是：

- 把 MemOS 改造成可服务 C 端产品的记忆后端
- 在其上加可控的 skill 执行层和 marketplace 能力

## 2. 分阶段路线图

### 阶段 0：基线收敛（1-2 周）

1. 固化现有 memo/payload/relation 数据模型。
2. 接入统一观测（请求、MCP 工具、SSE 事件）。
3. 补齐数据备份和迁移演练。

交付：可稳定运行的“内容记忆底座”。

### 阶段 1：长期记忆升级（2-4 周）

1. 新增 `memory_chunk` 与 `memory_embedding` 表。
2. 引入异步 embedding 任务流水线。
3. 新增 hybrid 检索 API（keyword + vector）。

交付：可语义召回的长期记忆服务。

### 阶段 2：Skill Runtime（3-6 周）

1. 定义 skill manifest/schema。
2. 上线执行器：
- code sandbox
- web search adapter
- file ops
- data analysis
3. 增加执行审计与配额控制。

交付：可执行你自定义 skills 的受控运行时。

### 阶段 3：Marketplace（4-8 周）

1. 技能发布/安装/版本管理。
2. 签名校验与安全扫描。
3. 评分、统计、灰度发布。

交付：可运营的 skills marketplace。

### 阶段 4：企业化治理（持续）

1. 数据合规（删除权、导出、脱敏）。
2. 成本治理（token/API/执行时长）。
3. 质量闭环（召回准确率、任务成功率、用户满意度）。

## 3. 风险清单

### 3.1 安全风险

- 代码执行 skill 造成越权/逃逸。
- 外部 API 密钥泄露。

控制：
- 强沙箱、网络白名单、短时凭证、全链路审计。

### 3.2 数据风险

- 错误记忆长期污染。
- 删除链路不完整引发合规问题。

控制：
- 记忆版本化、软删除 + 物理清理任务、纠错反馈通道。

### 3.3 成本风险

- embedding 与搜索调用成本失控。

控制：
- 分层索引（热点/冷数据）、批处理、缓存、配额。

### 3.4 质量风险

- 召回结果不准，skill 决策错误。

控制：
- Recall@K、正确率、失败分类、人工回放。

## 4. 关键 KPI（建议）

1. 记忆召回命中率（Recall@10）
2. 技能执行成功率
3. 平均响应延迟（P95）
4. 每日单位用户调用成本
5. 用户修正率（记忆错误需要人工修正比例）

## 5. 最终建议

先把 MemOS 用作“长期记忆与内容主库”，再叠加你自己的 skill runtime 与 marketplace，不建议直接在当前代码里硬塞全套 agent orchestration。
