## TL;DR

如果你只做三件事，优先做：

1. 复制 `MCP 工具治理`。
2. 复制 `CEL 过滤引擎`。
3. 复制 `双协议复用同一业务层`。

## 1. 高价值借鉴清单（按优先级）

| 优先级 | 技术点 | 借鉴价值 | 实施难度 | 代码锚点 |
|---|---|---|---|---|
| P0 | MCP 工具治理 | 快速形成可控 skill 网关 | 中 | `server/router/mcp/mcp.go` |
| P0 | CEL -> SQL 过滤引擎 | 统一过滤语言，兼容多数据库 | 中 | `internal/filter/*` |
| P0 | Connect + Gateway 复用业务内核 | 多客户端接入零重复业务 | 中 | `server/router/api/v1/v1.go` |
| P1 | 写后 payload 派生 | 为检索和推荐提供结构化索引 | 低 | `memopayload/runner.go` |
| P1 | JWT + PAT 双认证 | 同时覆盖用户态和程序态 | 中 | `server/auth/*` |
| P1 | migration embed 体系 | 升级可控、交付稳定 | 低 | `store/migrator.go` |
| P2 | SSE 可见性推送 | 实时前端/agent 更新体验 | 低 | `sse_hub.go` |
| P2 | 后台批处理 runner | 异步任务稳定性 | 低 | `server/runner/s3presign/runner.go` |

## 2. 每项技术怎么落到你项目

### 2.1 MCP 工具治理（最推荐）

落地动作：
1. 定义 `toolset` 分组。
2. 增加 `readonly` 全局开关。
3. 支持 include/exclude 覆写。
4. 执行前再次校验权限（不要只靠列表过滤）。

### 2.2 CEL 过滤引擎

落地动作：
1. 先定义你自己的字段 schema。
2. 保持三阶段：解析、IR、方言渲染。
3. 先做 20% 高频过滤语法，再扩。

### 2.3 双协议复用业务层

落地动作：
1. 协议处理器只做 request/response 适配。
2. 业务逻辑只留一份在 service 层。
3. 错误码统一映射，避免行为不一致。

## 3. 七天借鉴计划（可执行）

### Day 1-2

1. 在你项目建 MCP tool metadata 模型。
2. 接入 readonly/toolset/include/exclude。

### Day 3-4

1. 引入 CEL 过滤（先最小字段集）。
2. 在一个主列表接口试运行。

### Day 5-6

1. 打通双协议路由到同一 service。
2. 加统一鉴权（JWT + PAT）。

### Day 7

1. 加审计日志和 3 个核心指标（QPS/P95/错误率）。
2. 做一次压测和回归。

## 4. 不建议直接照搬的部分

1. 直接把 `search_memos` 当“真正全文检索”使用（目前更多是 contains/LIKE）。
2. 把内部 README 中的 “plugin” 语义误解成动态 marketplace 插件。

一句话：先借治理与架构模式，再按你的业务补执行层和语义能力。
