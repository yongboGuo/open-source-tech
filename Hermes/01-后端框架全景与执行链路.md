# Hermes 后端框架全景与执行链路

## 1. 结论先看
Hermes 的后端不是单一 Web 服务，而是一个“多入口 + 单 Agent 内核 + 可插拔工具体系 + 多形态持久层”的架构：

- 入口层：`cli.py`、`gateway/run.py`、`cron/scheduler.py`、ACP 适配层。
- 编排层：`run_agent.py` 的 `AIAgent` 负责主循环、模型调用、工具调度、重试与压缩。
- 工具层：`model_tools.py` + `tools/registry.py` 负责工具发现、schema 汇总、分发执行。
- 存储层：`hermes_state.py` 的 SQLite + FTS5（会话与全文检索）+ 文件型记忆（`MEMORY.md`/`USER.md`）。
- 扩展层：plugins（memory/context/tool/hook）、skills、skills hub。

该架构适合迁移到你自己的后端项目，尤其适合“一个核心 Agent 服务、多个接入渠道、可渐进扩容”的场景。

- 架构文档：`website/docs/developer-guide/architecture.md`

## 3. 后端分层拆解

### 3.1 入口层（Channels）
Hermes 同时支持 CLI、消息平台网关、定时任务，这些入口最终都会汇聚到 `AIAgent.run_conversation()`。

- CLI：交互式会话与 slash command。
- Gateway：平台消息事件 -> 会话识别 -> 命令拦截 -> agent 执行 -> 回传。
- Cron：调度器按 job 触发，支持绑定 skills。

### 3.2 Agent 编排层（核心）
`run_agent.py` 是事实上的 orchestration kernel：

- 构建系统提示词（身份、记忆、skills、上下文规则）。
- 调用模型 API（多 provider / 多模式）。
- 接收 tool calls 并循环执行。
- 做中断处理、预算控制、重试与上下文压缩。
- 写入会话存储，触发后置同步逻辑（memory sync、prefetch 等）。

### 3.3 工具运行时（Tool Runtime）
`model_tools.py` 做三件核心事：

- 启动时发现并注册工具（内置 + MCP + 插件）。
- 根据 enabled/disabled toolsets 生成给模型的可用工具 schema。
- 统一分发 tool call，并处理 sync/async 桥接。

这个层非常适合你借鉴成“技能/工具统一执行网关”。

### 3.4 存储层（State + Search）
`hermes_state.py` 的 `SessionDB` 关键点：

- SQLite WAL 模式，兼顾并发读与单写性能。
- `messages_fts`（FTS5）+ trigger 自动维护全文索引。
- 会话、消息、成本、tokens、标题、父子会话链路都落库。
- 写入采用应用层 jitter retry，避免 SQLite busy convoy。

### 3.5 Gateway 会话层
`gateway/session.py` 提供来源上下文与会话归一化：

- 统一多平台 session key。
- 可选 PII redaction。
- 多用户共享会话模式。
- 动态注入 session context 到系统提示词。

### 3.6 定时任务层
`cron/scheduler.py` 支持：

- RRULE 风格调度。
- 预执行脚本注入上下文。
- skills 绑定执行。
- 自动投递到来源/目标渠道。

## 4. 端到端执行链路

### 4.1 普通消息链路
1. 用户输入进入 CLI 或 Gateway。
2. 命令层先拦截（如 `/skills`、`/model`、`/cron`）。
3. 普通消息进入 `AIAgent.run_conversation()`。
4. 组装系统提示词 + 当前消息 + 可用 tools。
5. 模型返回 tool calls -> `model_tools.handle_function_call()` 分发。
6. 工具结果回注入消息，再次调用模型直到完成。
7. 结果写入 `SessionDB`，触发后置 sync（memory/session search 相关）。

### 4.2 skills 命令链路
1. `agent/skill_commands.py` 扫描技能目录并生成 `/skill-name` 命令映射。
2. 用户触发 `/xxx` 后，加载 skill 正文与 supporting files 索引。
3. skill 内容被组装成“系统激活块 + 正文 + 运行提示”注入本轮消息。
4. Agent 在统一工具体系下执行 skill 里建议的脚本、文件、搜索流程。

## 5. 借鉴到你项目的推荐形态

### 5.1 目标形态
- 一个 `Agent Runtime Service`（核心编排）。
- 多入口适配层（Web/API、IM bot、调度器）。
- 一个 `Tool Gateway`（工具 schema、权限、审计、执行）。
- 一个 `State Service`（PostgreSQL + 向量库 + 对象存储）。

### 5.2 为什么可复用
- Hermes 已经把“命令、技能、工具、会话、调度”抽象到可组合层次。
- 你可保留架构思想，不必照抄语言或目录结构。

## 6. 风险与边界
- `run_agent.py` 体量较大，迁移时要先做职责拆分，避免单文件巨石复制。
- SQLite 适合单节点/中小并发，集中服务化后建议迁 PostgreSQL。
- 技能执行包含命令执行能力，必须配套沙箱与策略引擎。

## 7. 分阶段落地路线（MVP -> Production）

### Phase 0（1 周）
- 抽象你项目的统一 `AgentRuntime` 接口。
- 定义 `ToolSchema`、`ToolCall`、`ToolResult` 协议。
- 先接入 3 个基础工具：`exec`、`web_search`、`file_ops`。

### Phase 1（2-3 周）
- 建立消息入口层（API + 调度器）。
- 建立会话落库（PostgreSQL），支持 message timeline。
- 接入最小 skills 机制（`skills_list` + `skill_view` + `/skill` 注入）。

### Phase 2（3-5 周）
- 接入沙箱（你已有 E2B）并做工具执行隔离。
- 上线安全扫描、安装审计、命令审批。
- 引入长期检索（session_search + summarization）。

### Phase 3（持续）
- marketplace、计费、版本治理、灰度发布。
- 监控体系（成功率、延迟、工具错误、成本、用户留存）。
