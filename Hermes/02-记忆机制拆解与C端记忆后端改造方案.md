# 记忆机制拆解与 C 端记忆后端改造方案

## 1. 现状结论
Hermes 当前的“记忆”其实是三层：

1. 内建短结构化记忆（文件型）：`tools/memory_tool.py` -> `MEMORY.md` + `USER.md`。
2. 外部记忆插件层：`agent/memory_provider.py` + `agent/memory_manager.py`。
3. 长期会话检索层：`tools/session_search_tool.py` + `hermes_state.py`（SQLite FTS5）。

这三层天然对应你做 C 端产品记忆后端的三类数据：
- 用户偏好事实（profile facts）
- 可检索的情景记忆（episodic memory）
- 可行动的策略/技能记忆（procedural memory）

## 2. 源码证据
- 文件记忆：`tools/memory_tool.py`
- 记忆管理器：`agent/memory_manager.py`
- provider 抽象：`agent/memory_provider.py`
- Agent 注入与同步：`run_agent.py`
- 长期检索：`tools/session_search_tool.py`
- 会话数据库：`hermes_state.py`

## 3. Hermes 记忆机制拆解

### 3.1 内建记忆（memory_tool）
`memory_tool.py` 的关键行为：

- 存储位置：`$HERMES_HOME/memories/MEMORY.md` 与 `USER.md`。
- 限额：字符级限制（默认 memory 2200，user 1375）。
- 操作：`add` / `replace` / `remove`。
- 持久化：原子写（temp + `os.replace`）+ 文件锁。
- 安全：对注入/外传模式做 pattern scan。

设计亮点：系统提示词使用“冻结快照”，中途写入不刷新当前系统提示词，减少 prompt cache 抖动。

### 3.2 外部 memory provider
`memory_provider.py` 定义了 provider 生命周期与 hook：

- `initialize` / `prefetch` / `sync_turn` / `queue_prefetch`
- `system_prompt_block` / `get_tool_schemas` / `handle_tool_call`
- `on_session_end` / `on_pre_compress` / `on_memory_write`

`memory_manager.py` 负责统一编排与容错。实现层面强调“最多一个 external provider”。

### 3.3 会话检索记忆（session_search）
`session_search_tool.py` 的链路：

- 先用 FTS5 搜消息。
- 按 session 去重并排除当前会话链。
- 拉取会话对话并截断窗口。
- 调用辅助模型做定向摘要。
- 返回总结而不是原始大文本。

这部分本质上就是“跨会话情景记忆召回”。

## 4. 可行性评估：能否改成 C 端产品记忆后端
结论：完全可行，而且 Hermes 已经给了最关键的接口抽象。

- 你不需要沿用文件存储。
- 你可以把 `MemoryProvider` 变成数据库 provider。
- `sync_turn + prefetch + session_search` 组合，天然就是“持续更新 + 按需召回”的记忆系统。

## 5. 建议目标架构（你的项目）

### 5.1 组件划分
- `memory-write-service`：处理每轮对话后的事实抽取与写入。
- `memory-query-service`：负责预取与召回排序。
- `memory-profile-service`：用户长期偏好画像。
- `session-recall-service`：跨会话检索与摘要。

### 5.2 存储建议
- 结构化事实：PostgreSQL。
- 向量检索：pgvector 或独立向量库。
- 原始会话：PostgreSQL + 对象存储（按策略归档）。

## 6. 数据模型建议（最小可用）

```sql
-- 用户画像事实
create table user_memory_fact (
  id bigserial primary key,
  user_id text not null,
  fact_key text not null,
  fact_value text not null,
  confidence numeric(4,3) default 0.7,
  source_session_id text,
  version int default 1,
  created_at timestamptz default now(),
  updated_at timestamptz default now()
);

-- 情景记忆（可向量召回）
create table episodic_memory (
  id bigserial primary key,
  user_id text not null,
  session_id text not null,
  role text not null,
  content text not null,
  embedding vector(1536),
  created_at timestamptz default now()
);

-- 记忆写入审计
create table memory_audit_log (
  id bigserial primary key,
  user_id text not null,
  action text not null,
  payload jsonb not null,
  created_at timestamptz default now()
);
```

## 7. API 契约建议

- `POST /memory/sync-turn`
- `POST /memory/prefetch`
- `POST /memory/write-fact`
- `POST /memory/session-search`
- `POST /memory/feedback`（用户纠错）

把这些对齐到 Hermes 的 provider 生命周期即可无缝迁移思路。

## 8. 持续更新机制（关键）

每个 turn 做 4 步：
1. 对话结束 -> 抽取候选事实（偏好、约束、长期目标）。
2. 去重与冲突检测（同 key 新旧事实冲突）。
3. 写入事实表并版本化。
4. 异步向量化，供下轮 prefetch / session search。

可叠加用户反馈回路：
- 用户“记错了/别这么做” -> 直接触发 fact replace。

## 9. 风险与边界
- 记忆污染：错误事实写入会长期影响输出。
- 隐私合规：C 端必须支持删除权、导出权、数据最小化。
- 成本：每轮向量化与摘要会增加 token+算力成本。
- 一致性：并发写入需要乐观锁或版本字段。

## 10. 分阶段落地

### MVP（2 周）
- 先实现结构化用户偏好记忆，不做向量库。
- 实现 `sync_turn` 与 `prefetch`。
- 支持用户侧“纠错覆盖”。

### Beta（3-5 周）
- 上线 session_search（全文检索 + 摘要）。
- 引入 embedding + 相似召回。
- 建立记忆审计与可观测指标。

### Production（持续）
- 上线隐私治理（删除、导出、TTL）。
- 引入质量控制（事实冲突检测、置信度衰减）。
- 支持多租户与分层缓存。
