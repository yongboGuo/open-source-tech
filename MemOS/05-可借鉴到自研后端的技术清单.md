## 1. 可直接借鉴（高优先级）

### 1.1 双协议 API 复用同一业务层

来源：`server/router/api/v1/v1.go` + `connect_handler.go`

做法：
- gRPC-Gateway 和 Connect 都代理到同一个 `APIV1Service`。

收益：
- 多客户端接入（浏览器/SDK/CLI）时不重复业务实现。

### 1.2 CEL 过滤编译为多数据库 SQL

来源：`internal/filter/*`

做法：
- `CEL -> IR -> SQL`，并按 sqlite/mysql/postgres 渲染差异。

收益：
- 过滤语言统一、数据库可替换、业务不改查询语义。

### 1.3 MCP 工具治理模型

来源：`server/router/mcp/mcp.go` + `tool_metadata.go`

做法：
- toolset/readOnly/include/exclude + 服务端二次校验。

收益：
- 适合作为 skills 网关权限治理模板。

### 1.4 持续派生字段（payload）机制

来源：`memo_service.go` + `memopayload/runner.go` + `internal/markdown/markdown.go`

做法：
- 写入内容时重建 tags/property/mentions。

收益：
- 为检索、通知、排序提供结构化索引，且持续更新。

### 1.5 SSE 实时事件通道

来源：`sse_hub.go`

做法：
- 按可见性/角色过滤推送，慢客户端丢弃事件避免阻塞。

收益：
- 前端和 AI 客户端可实时感知记忆变更。

## 2. 也值得借鉴（中优先级）

### 2.1 统一认证器（JWT + PAT + Refresh）

来源：`server/auth/token.go` + `authenticator.go`

亮点：
- access token 短期、refresh token 长期、PAT 程序化接入。
- PAT 哈希存储 + 最后使用时间回写。

### 2.2 内嵌 migration + 版本治理

来源：`store/migrator.go`

亮点：
- `embed` SQL + LATEST 快速初始化 + 增量迁移事务化。

### 2.3 轻量缓存层

来源：`store/cache/cache.go`

亮点：
- TTL + 清理协程 + 数量阈值驱逐。

### 2.4 后台 batch runner 模式

来源：`server/runner/s3presign/runner.go`

亮点：
- 分批处理 + 周期调度 + 可中断上下文。

## 3. 可借鉴但要注意边界

### 3.1 `search_memos` 标称 Full-text，实际为 contains/LIKE

来源：`server/router/mcp/README.md` 与 `tools_memo.go` + `internal/filter/render.go`

建议：
- 如果用于 C 端记忆检索，尽快上 FTS 或向量索引，不要停留在 LIKE。

### 3.2 “Plugin”在 MemOS 中多为内部模块语义

来源：`internal/email/README.md`、`internal/scheduler/README.md`

建议：
- 把它理解为内部组件，不是动态 marketplace 插件系统。

## 4. 可复用到你后端项目的最小组合

建议优先复制这 5 件事：

1. `APIV1Service` 式协议与业务解耦。
2. CEL 过滤引擎。
3. MCP 工具治理模型。
4. payload 派生字段更新流水线。
5. 统一认证器（JWT+PAT）。

这 5 件能快速搭好“可被 Agent 使用的记忆后端底座”。
